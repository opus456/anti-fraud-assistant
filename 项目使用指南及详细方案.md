## 2. 使用与部署指南总览

如果你的目标是“把项目跑起来并真正用起来”，优先看这一章。这一章把本项目最常见的几种使用方式集中到一起，包括：

### 2.1 使用方式总览

这个项目主要有三种使用方式：

1. 本地开发模式：分别启动数据库、Python 后端、Node 网关和 React 前端
2. Docker 基础模式：只启动 PostgreSQL 和 Redis，应用服务本机运行
3. Docker 完整模式：前端、网关、后端、数据库、Redis 全部容器化运行

如果你是开发调试，建议使用第 1 种。

如果你是演示答辩或交付部署，建议使用第 3 种。

### 2.2 本地开发使用指南

在项目根目录按顺序执行：

```bash
docker compose up -d

cd server-python
pip install -r requirements.txt
python scripts/init_knowledge.py
python run.py

cd ../server-node
npm install
npm start

cd ../client-react
npm install
npm run dev
```

启动完成后访问：

- 前端：http://localhost:5173
- Python 后端：http://localhost:8000
- Python 文档：http://localhost:8000/docs
- Node 网关：http://localhost:3001

这种方式适合：

- 本地开发
- 联调接口
- 演示单个模块
- 排查前后端问题

### 2.3 Docker 基础部署指南

如果你只想快速准备数据库和缓存，而应用服务继续在本机运行，可以使用：

```bash
docker compose up -d
```

这个命令会启动：

- PostgreSQL，端口 5432
- Redis，端口 6379

之后你仍然需要手动启动：

- server-python
- server-node
- client-react

这种方式适合：

- 日常开发
- 数据库调试
- 不希望每次都重建应用容器的场景

### 2.4 Docker 完整部署指南

如果你希望整个系统一次性容器化启动，使用：

```bash
docker compose -f docker-compose.prod.yml up -d --build
```

完整部署会启动以下服务：

- postgres
- redis
- python-backend
- gateway
- frontend

默认访问入口：

- 前端：http://localhost:80
- Node 网关：http://localhost:3001
- Python 后端：http://localhost:8000
- Python 健康检查：http://localhost:8000/health
- Node 健康检查：http://localhost:3001/health

常用运维命令：

```bash
docker compose -f docker-compose.prod.yml ps
docker compose -f docker-compose.prod.yml logs -f
docker compose -f docker-compose.prod.yml down
```

这种方式适合：

- 比赛演示环境
- 答辩现场部署

## 3. 项目概述

“破局·御见”是一个多模态 AI 反诈检测系统，面向个人用户、监护人和管理员，提供从输入检测、风险判定、告警干预到知识库更新的完整闭环。

系统的核心定位不是单一模型分类器，而是一个工程化反诈平台。它将规则引擎、RAG 检索、大模型推理、用户画像、记忆机制和实时告警整合到同一套业务流程中。

当前项目已经支持：

- 文本检测
- 图片 OCR 后检测
- 音频转写后检测
- 多模态融合检测
- 知识库搜索与维护
- 监护人联动与实时预警
- 管理员端知识维护与统计查看

## 5. 系统总体方案

### 5.1 四层架构

当前项目采用“感知 - 决策 - 干预 - 进化”的四层架构。

#### 1) 感知层

负责接收多种输入：

- 文本输入
- 图片上传
- 音频上传
- 多模态组合输入

其中图片通过 OCR 提取文本，音频通过语音转写提取文本，多模态请求会将文本、图片 OCR、音频转写、视频描述、上下文统一汇总后进入主检测链路。

#### 2) 决策层

负责完成诈骗识别与风险计算，核心流程为：

1. 关键词规则引擎初筛
2. RAG 检索相似诈骗案例
3. 条件触发或全量调用 LLM CoT 推理
4. 多源融合与一致性校正
5. 个性化风险评估

#### 3) 干预层

由 Node.js 网关承担实时联动职责，负责：

- WebSocket 连接管理
- API 代理
- 高风险结果广播
- 用户端与监护人端房间通知

#### 4) 进化层

负责让系统持续增强，当前真实落地方式为：

- 管理员确认新型诈骗案例
- 写入知识库
- 生成 embedding
- 纳入后续 RAG 检索体系

这意味着当前“进化”本质上是知识库和检索能力持续增强，而不是在线训练基础模型参数。

## 8. 核心算法与判定方案

### 8.1 文本检测主流程

当前文本检测的核心路径为：

1. 文本清洗
2. 关键词风险检测
3. 紧迫词、高危模式、组合信号抽取
4. RAG 查询相似诈骗案例
5. 必要时调用 LLM 做结构化语义判断
6. 多源融合输出最终风险
7. 结合用户历史风险和画像做个性化修正
8. 生成风险等级和干预动作

### 8.2 规则引擎

规则引擎承担第一层低成本高召回筛查职责。

它会提取以下信息：

- 诈骗类别关键词
- 紧迫信号
- 高危正则模式
- 链接、手机号、卡号等特征信号
- 权威机构 + 转账、远程控制 + 凭证信息等组合风险
- 安全提示语境与误报抑制信号

### 8.3 RAG 检索增强

知识库存储在 PostgreSQL 的向量表中，使用 pgvector 和 HNSW 索引支持近似最近邻检索。

当前关键实现参数：

- embedding 维度：1024
- HNSW m = 16
- ef_construction = 64

RAG 的作用不是直接给结论，而是：

- 找相似案例
- 补全诈骗类型
- 对规则分数进行抬升
- 为 LLM 提供上下文

### 8.4 LLM CoT 推理

LLM 在项目中的职责是对复杂语义、高隐蔽样本、规则与检索冲突样本做结构化判断。

输出内容通常包括：

- risk_level
- scam_type
- reason
- urgency_check
- financial_check
- authority_fake_check
- info_theft_check
- too_good_check

### 8.5 个性化风险评估

当前实现并不是真实的 LSTM 网络，而是“时序衰减 + 用户画像权重”的轻量模型。

核心输入包括：

- 当前内容风险
- 最近 30 天历史风险
- 角色类型
- 年龄特征

这使得同样一段文本在不同用户身上可以得到不同的最终风险强度。

## 9. 多模态方案说明

### 9.1 图片检测

接口会接收图片文件，进行 OCR 分析后，将提取出的文本作为检测输入。

当前实现特点：

- 支持 JPG、PNG、GIF、WebP
- 文件大小有限制
- 先 OCR，再走文本反诈检测
- 为了节约时延，图片检测默认可跳过 LLM CoT

### 9.2 音频检测

接口会接收音频文件并进行转写，然后再进入同一主检测链路。

当前支持：

- MP3
- WAV
- M4A
- OGG
- WebM

### 9.3 多模态融合检测

多模态请求支持合并：

- text
- image_ocr
- image_frame
- audio_transcript
- video_description
- context

本质上当前项目的多模态在线推断方案是“先转文本语义，再做统一反诈判断”。

优点：

- 结构清晰
- 工程复杂度低
- 复用单一主链路
- 方便退化运行

## 12. 典型使用流程

### 12.1 用户自检流程

1. 用户登录系统
2. 在检测页面输入文本或上传图片、音频
3. 系统调用 Python 检测接口
4. 后端执行规则引擎、RAG、LLM、个性化评分
5. 前端展示风险等级、诈骗类型、原因解释与建议

### 12.2 高危告警流程

1. 后端检测出高危结果
2. Node 网关接收告警信息
3. 广播到用户房间和监护人房间
4. 用户端弹窗提醒
5. 监护人端同步收到联动预警

### 12.3 知识进化流程

1. 管理员识别新型诈骗案例
2. 添加或导入知识条目
3. 系统生成 embedding 并写入知识库
4. 后续相似文本会被 RAG 召回
5. 检索能力随知识增长而增强
